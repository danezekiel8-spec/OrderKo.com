# OrderKo Render Deployment Runbook

This runbook is for OrderKo production or pilot deployment on Render with Render Postgres.

## Deployment Shape

- Web service: Node / Next.js
- Health check: `/api/health`
- Production database: Render Postgres
- Runtime schema: `prisma/postgres/schema.prisma`
- Local development schema: `prisma/schema.prisma` with SQLite

The local SQLite setup is intentionally kept for fast development. Render must use the Postgres scripts in `render.yaml`.

## Required Render Environment Variables

- `DATABASE_URL`: supplied by the Render Postgres database
- `NODE_ENV`: `production`
- `NODE_VERSION`: `20.11.1` or newer Node 20
- `STAFF_SESSION_SECRET`: generated by Render or manually supplied, at least 32 characters
- `STAFF_PIN_SECRET`: generated by Render or manually supplied, at least 32 characters
- `ORDERKO_SUPER_ADMIN_SECRET`: generated by Render or manually supplied, at least 32 characters
- `ORDERKO_DEFAULT_RESTAURANT_SLUG`: optional default username for staff login
- `ORDERKO_QR_BASE_URL`: final HTTPS domain, for example `https://orderko.com`
- `CLOUDINARY_CLOUD_NAME`
- `CLOUDINARY_API_KEY`
- `CLOUDINARY_API_SECRET`
- `SMTP_USER`, `SMTP_PASS`, `LEAD_NOTIFICATION_EMAIL`, `LEAD_EMAIL_FROM`: optional but recommended for demo request emails

Run locally before deployment:

```bash
npm run env:check:production
```

For local checks against `.env`, use:

```bash
npm run env:check
```

## Build And Migration Commands

Render Blueprint build command:

```bash
npm ci && npm run build:render
```

Start command:

```bash
npm run start:render
```

The start command runs Prisma `migrate deploy` against the Postgres schema before serving traffic. Create restaurants from `/super-admin`; production startup should not seed a sample restaurant.

## Smoke Test

After deployment:

```bash
ORDERKO_SMOKE_BASE_URL=https://your-render-url.onrender.com npm run test:prod-smoke
```

This smoke test is read-only. It checks health, the public menu, manifest, and staff route protection.

## QR Domain Check

Before printing QR codes:

1. Set `ORDERKO_QR_BASE_URL` to the final HTTPS domain.
2. Regenerate the QR in admin.
3. Scan with iPhone Safari and Android Chrome.
4. Confirm the QR opens `/r/[slug]`, not localhost or a LAN IP.

## Launch-Day Rollback

If a deployment breaks ordering:

1. Pause ordering in admin if the app is still reachable.
2. Switch staff to paper order fallback.
3. Check Render service logs and `/api/health`.
4. Roll back to the last successful Render deploy if the issue is code-related.
5. If the issue is environment-related, fix the env var and redeploy.
6. Place one test order after recovery, then resume QR ordering.
